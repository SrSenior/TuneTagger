# Development Notes

This document will be used to record the development process of TuneTagger.

Since this is my first project using .NET and ASP.NET Core Web API, the content will keep changing as the project progresses. The goal is not only to document what gets implemented in each commit, but also to write down things I learn or re-learn during the process.

This document also works as a personal learning log. Because of that, some explanations may seem obvious to more experienced developers, but I include them because they help me understand the technical decisions behind the project and give me material I can revisit later.

The language used here will not necessarily be the most technical possible, but I will try to keep it clear, organized and useful. The intention is for this document to work both as personal documentation and as a general reference for the progress of the project.

I will also maintain a Spanish version of this document for my own reference and for Spanish-speaking readers.

## General Project Context

TuneTagger is an application designed to analyze local audio files, identify the actual song, and return useful metadata such as title, artist, album, confidence score and a suggested file name.

The main idea behind the project comes from a common problem when managing downloaded or locally stored MP3 files: many files have unclear, incomplete or incorrect names. For example, a file might be named something like `Black Clover Opening 10 Full.mp3`, even though the actual song has a specific artist and title.

The initial goal of TuneTagger is to support this flow:

```txt
local audio file
→ fingerprint generation
→ AcoustID lookup
→ best match selection
→ clean metadata response
→ suggested file name
```

For now, the project is being built as a local-first application. This means the app is intended to run locally, not as a public platform where users upload files to an external server.

## Initial Backend State

To begin, an API was implemented using ASP.NET Core Web API. This API listens for requests on `localhost`, which means that during development it only runs on the local machine.

At first, basic endpoints were created to confirm that the API was working correctly:

* `GET /api/health`: a test endpoint used to confirm that the backend is running.
* `GET /api/tracks/mock-analysis`: a test endpoint that returns mocked song data.
* `POST /api/tracks/analyze`: the main endpoint used to receive and analyze an audio file.

A model named `TrackAnalysisResult` was also created. This model represents the final response of a track analysis and includes data such as:

* original file name;
* detected title;
* artist;
* album;
* suggested file name;
* match confidence;
* analysis status.

Using models helps keep API responses clearer and more structured, instead of returning anonymous objects everywhere.

## 24/06

`fpcalc.exe` was added to the development flow.

`fpcalc` is a Chromaprint tool used to generate an acoustic fingerprint from an audio file. This fingerprint represents the sound content of the file, not just its file name or existing metadata.

The fingerprint generated by `fpcalc`, along with the song duration, is later used to query AcoustID and search for matches in its database.

A README file was added inside:

```txt
/backend/TuneTagger.Api/Tools
```

That README documents important details about `fpcalc`, especially its expected location inside the project and how to configure it for local development.

The backend `.csproj` file was also modified so that `fpcalc.exe` is copied to the output directory during the build process.

This is necessary because, when the project runs, .NET does not execute directly from the project root. Instead, it runs from an output folder such as:

```txt
backend/TuneTagger.Api/bin/Debug/net10.0/
```

Because of that, if `fpcalc.exe` only exists in the original `Tools` folder, the application might not find it at runtime. To avoid that issue, the project was configured to copy it to:

```txt
backend/TuneTagger.Api/bin/Debug/net10.0/Tools/fpcalc.exe
```

This allows the project to use relative paths during development and avoids depending on an absolute path from my own computer.

## 25/06

Configuration for AcoustID was added.

The AcoustID base URL was placed in `appsettings.Development.json`, because it is public configuration and may change depending on the environment.

However, the AcoustID API key was not placed directly in any configuration file in the repository, because it is sensitive information that should not be pushed to GitHub.

For local development, the decision was made to use `dotnet user-secrets`. This tool allows development secrets to be stored outside the repository, reducing the risk of accidentally exposing private keys.

The general separation is:

* public configuration: `appsettings.Development.json`;
* local secrets: `dotnet user-secrets`;
* Docker or production in the future: environment variables.

This separation is important because it avoids mixing normal configuration with sensitive information.

## 26/06

The backend was connected to AcoustID.

The main endpoint no longer only receives the file. It now performs the real analysis flow:

```txt
received file
→ temporary file save
→ fpcalc execution
→ duration and fingerprint extraction
→ AcoustID lookup
→ JSON response reading
```

The POST request to AcoustID was also implemented using GZip compression. This was done because fingerprints can be quite long, and AcoustID recommends compressing this type of request.

The request sent to AcoustID includes data such as:

* `client`: the application API key;
* `duration`: the duration of the audio file;
* `fingerprint`: the fingerprint generated by `fpcalc`;
* `meta`: the metadata requested from AcoustID;
* `format`: the response format, in this case JSON.

The response from AcoustID also started being processed in order to select only the best match, using the `score` value returned by the API.

The `score` represents how strong the match is. Because of that, if AcoustID returns multiple results, the app selects the result with the highest confidence.

## 27/06

Code modularization started.

Up to this point, most of the logic was inside `Program.cs`. This worked well for quickly validating the flow, but it made the file grow too much and mixed several responsibilities in one place.

The decision was made to move part of the logic into services.

The main idea was to divide the backend like this:

```txt
Program.cs
→ defines endpoints and HTTP responses

FingerprintService
→ runs fpcalc and gets duration + fingerprint

AcoustIdService
→ queries AcoustID and gets the best match
```

This separation allows each class to have a clearer responsibility.

`FingerprintService` only handles `fpcalc`. It does not need to know anything about AcoustID or HTTP responses.

`AcoustIdService` only handles communication with AcoustID. It does not need to know how the file was uploaded or how the fingerprint was generated.

`Program.cs` remains the place where HTTP requests are received, services are called, and responses are returned to the client.

A `TrackAnalysisService` was also considered as a way to orchestrate the whole analysis flow, but for now the decision was made to keep that coordination in `Program.cs` to avoid adding an extra layer too early.

## 28/06

The main AcoustID logic was modularized.

Inside `AcoustIdService`, internal responsibilities were separated into smaller methods. In particular, the GZip compression logic and the logic used to extract the best match from the AcoustID results were separated.

This makes the main method easier to read, since it no longer needs to contain every technical detail in one large block.

The general structure became:

```txt
FindBestMatchAsync
→ prepares data
→ creates compressed request
→ sends request to AcoustID
→ validates response
→ gets results
→ extracts best match
```

It was also decided that `AcoustIdService` should return `null` when AcoustID does not find a useful match. This is better than returning a fake result with values such as `"Unknown"` or `"No match found"`.

The logic becomes clearer:

```txt
AcoustIdBestMatch
→ a useful match was found

null
→ no useful match was found

throw
→ a technical error happened
```

Examples of technical errors include:

* missing AcoustID base URL;
* missing API key;
* AcoustID returning an HTTP error;
* the JSON response not having the expected format.

With this, `Program.cs` can better distinguish between an unrecognized audio file and a real system error.

The final analysis status was also updated to use a clearer value such as `matched` when a match is found.

## End of Stage A: Functional Backend

With these changes, Stage A of the roadmap is considered complete.

The goal of Stage A was to make the backend able to analyze a single song in a functional way.

The backend can now:

* receive an audio file through `POST /api/tracks/analyze`;
* validate allowed extensions;
* temporarily save the uploaded file;
* run `fpcalc`;
* get `duration` and `fingerprint`;
* query AcoustID;
* compress the POST request using GZip;
* process the JSON response;
* select the match with the highest confidence;
* return a clean response to the frontend;
* generate a `suggestedFileName`;
* delete the temporary file after analysis.

This does not mean the backend is completely finished, but it does mean that the main MVP flow is working.

Possible improvements remain for later, such as:

* more detailed error handling;
* using `IHttpClientFactory`;
* basic tests;
* batch processing;
* writing ID3 tags;
* real file renaming;
* job queue integration with RabbitMQ;
* Docker packaging.

However, none of those improvements block the start of the next phase.

## Stage B: Functional Frontend

After completing the main backend flow, Stage B of the roadmap was started.

The goal of this stage is to build a local web interface that allows users to interact with TuneTagger without needing to use `curl` or test the API manually from the terminal.

The frontend will be built with:

* Vite;
* React;
* TypeScript;
* pnpm as the package manager;
* Tailwind CSS for styling.

The frontend project was created with Vite using the React + TypeScript template. The decision was also made to use `pnpm` instead of `npm` as the package manager.

The initial frontend structure was created inside:

```txt
frontend/tunetagger-web
```

Tailwind CSS was also installed and configured correctly with Vite. The initial test confirmed that Tailwind works, since utility classes could be applied directly in `App.tsx`.

For now, the initial frontend goal will be kept simple:

```txt
select file
→ press analyze button
→ send file to backend
→ show analysis status
→ show result
```

The expected initial features are:

* screen for selecting an audio file;
* button to analyze the file;
* current status display;
* handling states such as `idle`, `analyzing`, `matched`, `not-found` and `error`;
* displaying title, artist, album, confidence and suggested file name;
* simple, local and functional interface.

The priority of this stage is not to create a visually perfect interface from the beginning, but to make sure the frontend connects correctly to the backend and allows the main TuneTagger flow to be used from the browser.
